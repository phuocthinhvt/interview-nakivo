# Question A - System Design

To support 1 million jobs smoothly, we apply Event-Driven Architecture using a dedicated message broker (Apache Kafka or RabbitMQ) with a high-speed distributed cache (Redis)

## Architecture:

API Gateway / Load Balancer: balances incoming POST /api/jobs requests across application instances :

Producers: Receive HTTP requests, persist the initial state to MySQL, and immediately publish a lightweight message to the message broker.
Consumers : Listen to the message broker, pull message, and process message asynchonously using configured thread pools.
Message Broker : Acts as the data pipeline. It handles sudden traffic, guarantees message delivery, and separates receiving from processing.
Redis Cluster: Acts as an very high-speed, distributed locking and duplicate prevention system. 

## Data Flow

Client send POST /api/jobs. The receiving App Instance generates a unique jobId, saves the record to MySQL with status PENDING and publishes a small payload to a Kafka topic.
Controller returns 202 Accepted instantly to the client
Kafka distributes the messages across partitions. Each consumer instance is assigned specific partitions inside a shared Consumer Group.
A worker instance consumes the message, acquires a distributed lock in Redis for jobId, fetches the latest data from MySQL, executes the business logic, and transitions the status to COMPLETED or handles failures.

## Duplicate Processing Prevention

Distributed Locking with Redis
Before a worker process starts executing a job, it must acquire a distributed lock using Redis.

Optimistic Locking in DB
Once the lock is acquired, the worker opens a database transaction and checks the current status:
If the status is already PROCESSING, COMPLETED, or FAILED, the worker releases the lock and skips processing.
If it is PENDING, it update the status to PROCESSING immediately.

## Failure Handling & Retry Strategy

If a job fails, the worker handles the error, increases the retry count in MySQL, saves the error message, and sends the job to a waiting queue with a timer (TTL) before trying again.
The job sits in the retry queue for an increasing duration before being pushed back into the Main Queue.This gives downstream systems time to recover.
After 3 failures, the job is marked as FAILED in MySQL and moved to a Dead Letter Queue (DLQ). This stops the endless retry loop and isolates bad data.

## Scaling Approach

Application Scaling: App instances do not store data locally. We can automatically add more instances via Kubernetes.
Broker Scaling: In Kafka, we set the number of partitions to match or exceed our maximum expected worker threads. This ensures multiple instances can process messages at the same time without blocking each other.
Database Scaling: Our workers mostly perform write actions. All dashboard and monitoring requests (GET /api/jobs) will be routed to MySQL Read Replicas to stop slowing down active jobs.

## Operational Considerations

Consumer Lag Monitoring: If lag increases continuously, it is a signal to add more app instances.
Trace ID Logging: Inject a unique ID at the API gateway and pass it through Kafka headers down to the worker thread. If a job fails, we can search that ID in Kibana/Grafana to see the complete history from the initial HTTP POST down to the database error.
Alerting on DLQ: Set an automated alert on the Dead Letter Queue. If messages in the DLQ spike quickly, it means a major system error has occurred, so the team can fix it right away. 

# Question B - Database Performance

## Investigation Steps
* Extract the exact SQL query generated by Spring Data JPA under the hood.
* Run `EXPLAIN ANALYZE` on the target query to check the join/scan types. Look for `Table scan` (indicating a full table scan) and `Sort/Filesort` operations (indicating manual sorting in memory/disk).
* Inspect Monitor tools to identify hardware bottlenecks: whether the system is Disk I/O bound or CPU bound.

## Likely Bottlenecks at 50M Rows
The `status` column has very few distinct values. If `PENDING` jobs accumulate significantly, MySQL's optimizer will determine that a full table scan is cheaper than hopping back and forth via a single-column index, causing severe disk I/O bottlenecks.
Sorting by `created_at DESC` without a matching index forces MySQL to load all filtered records into `sort_buffer_size`. At this scale, the dataset easily overflows the RAM allocation and spills to disk, executing an extremely slow disk-based merge sort.
Returning a Spring Data `Page<Job>` automatically fires a synchronous `SELECT COUNT(*)` query alongside the pagination query. Scanning millions of index keys to compute an exact total count on every API call completely paralyzes performance.

## Proposed Database & Query Changes
Create a single combined index on `(status, created_at DESC)`. By placing the filter column first and the sorting column second, MySQL fetches pre-sorted data instantly, eliminating the `Filesort` overhead.
Change the repository return type from `Page<Job>` to `Slice<Job>`. This completely cuts out the hidden `COUNT` query, executing only a clean `LIMIT size + 1` query to fetch data.

## Technical Trade-offs
Introducing a large composite index increases storage consumption and adds a minor overhead to `INSERT` and `UPDATE` operations, as the B-Tree must be rebalanced.
Switching to `Slice<Job>` replaces traditional page numbers (1, 2, 3...) with an infinite-scroll or "Load More" navigation model on the frontend client.
